# 核心知识：PMCA团队协作组件选型策略

本文档是 `PMCATeamDecision` 智能体进行团队协作模式选择的核心决策依据。在分析任务时，必须严格遵循以下原则。

### 1. `RoundRobin` (轮询模式)

- **核心特征**: 线性、有序、可预测。智能体按预设顺序依次执行，如同工厂流水线。
- **适用场景**:
  - **线性流水线任务**: 任务步骤固定，前后依赖关系明确（例如：数据提取 -> 数据处理 -> 数据上报）。
  - **提议-批判循环**: 两个智能体（如“方案设计者”和“方案评审员”）之间进行多轮迭代。
  - **简单序列对话**: 需要多个专家按顺序对同一主题发表意见。
- **决策关键**: 当任务可以被清晰地分解为一系列**连续的、无分支的**步骤时，应优先选择 `RoundRobin`。
- **反模式**: **不要**在需要复杂条件分支、并行处理或动态调整流程的任务中使用 `RoundRobin`，这会导致效率低下和逻辑僵化。

### 2. `Swarm` (蜂群模式)

- **核心特征**: 灵活、动态、去中心化。任务在智能体之间自由流转，由最合适的“专家”主动承接。
- **适用场景**:
  - **探索性与研究性任务**: 任务目标明确，但实现路径不确定（例如：“调研并总结一个未知领域的最新进展”）。
  - **需要动态规划**: 任务的下一步依赖于上一步的执行结果，无法预先规划完整流程。
  - **突发事件处理**: 应对需要多领域专家协同诊断的复杂问题（例如：“分析一个紧急的线上Bug”）。
- **决策关键**: 当任务具有**高度不确定性**，需要智能体根据当前上下文自主决定下一步行动和协作对象时，`Swarm` 是最佳选择。
- **反模式**: 对于有固定业务流程、需要严格保证执行顺序和结果可复现的任务，**避免**使用 `Swarm`，其高度的灵活性可能导致流程不可控。

### 3. `GraphFlow` (图流程模式)

- **核心特征**: 结构化、可控、支持复杂逻辑。通过预定义的有向图来控制智能体之间的协作路径，支持并行、循环和条件分支。
- **适用场景**:
  - **复杂业务流程 (BPM)**: 需要稳定复现的、标准化的业务流程（例如：“处理一笔新的用户订单”，包含库存检查、支付、发货等多个并行或串行节点）。
  - **有明确分支和依赖的任务**: 任务中包含清晰的 "if-then-else" 逻辑。
- **决策关键**: 当任务流程可以被**形式化地描述为一个流程图**，并且需要高可靠性和可复现性时，应选择 `GraphFlow`。
- **反模式**: **不要**为简单的线性任务或高度探索性的任务设计复杂的 `GraphFlow`，这会带来不必要的开发和维护成本。

### 4. `MagenticOne` (磁力一号模式)

- **核心特征**: 中心化协调、强大的外部交互。由一个“协调员”智能体统一规划，并调用其他工具型智能体（如文件、网络交互）来完成任务。
- **适用场景**:
  - **与外部世界交互的开放性问题**: 需要从文件系统、网页、API等外部源获取信息并进行处理的任务。
  - **需要多工具协同**: 任务需要一系列不同工具的组合才能完成（例如：“从网站下载数据，存入本地文件，然后进行分析”）。
- **决策关键**: 当任务的核心是**与外部环境进行信息交互**，并且可以通过一个中心化的大脑进行规划时，`MagenticOne` 是合适的。
- **反模式**: 对于纯粹内部计算或智能体之间认知协作的任务，`MagenticOne` 的中心化协调员可能会成为瓶颈，此时 `Swarm` 或 `RoundRobin` 可能更高效。
